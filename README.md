```bash
# Frontend deps
cd frontend && npm install && cd ..
```

## Regenerate the corpus + graph

The corpus is generated by seeded scripts, not hand-typed one file at a time. If you edit any
`scripts/generate_*.py`, **always regenerate from scratch** — the bulk generator's random draws are
order-dependent, so a partial re-run can leave orphaned files from a previous version sitting next
to the new ones:

```bash
bash scripts/regenerate_corpus.sh   # wipes + rebuilds corpus/02-07
python3 graph/build.py              # ingests corpus/ -> data/graph.pkl
```

## Run it

Quickest path — starts both servers in the background, builds the graph first if it's missing:

```bash
bash scripts/dev.sh    # starts backend :8123 + frontend :5173
bash scripts/stop.sh   # stops both
```

Or manually, in two terminals:

```bash
# terminal 1 — backend (port 8123 in this environment; change freely)
uvicorn backend.main:app --port 8123

# terminal 2 — frontend
cd frontend && VITE_API_BASE=http://localhost:8123 npm run dev -- --port 5173
```

Open `http://localhost:5173`. Four tabs: Briefings, Ask, Explorer, Retirement.

## Run the acceptance tests

```bash
python3 tests/acceptance_tests.py
```

Tests 5, 8, 9 need the backend running on `:8123`; everything else runs directly against
`data/graph.pkl`. All 10 acceptance tests from the build brief pass as of this commit.

## Environment variables

- `OPENAI_API_KEY` — optional. Without it, the briefing/query/retirement agents render from
  deterministic fallback templates instead of live model calls (see `graph/llm.py`). Every
  acceptance test and the demo script were verified to work without this key. Set it to see actual
  generated prose instead of the templates. Model id is the single `MODEL` constant in
  `graph/llm.py` (`gpt-4o` by default) — change it to whatever model your account has access to.
- `VITE_API_BASE` — frontend env var pointing at the backend URL (see Run It above).

## Known scope decisions (stated plainly, not hidden)

- **Regulatory docs are now correctly numbered and topically real** (OISD-105 Work Permit System,
  OISD-106 Pressure Relief & Disposal Systems, OISD-116 Fire Protection, OISD-118 Layouts, Factories
  Act 1948 §7A/§41C/§87), assembled from publicly available search-engine-indexed excerpts of each
  standard — not a full verbatim transcription of the official PDF, since it couldn't be fetched
  directly in this environment. Verify exact clause/sub-section numbers against
  https://www.oisd.gov.in/en-in/oisd-standards-list before a compliance audit relies on exact
  wording (`corpus/07_regulatory/`, `scripts/generate_bulk_corpus.py`). Also note the Factories Act,
  1948 has been substantially superseded by the Occupational Safety, Health and Working Conditions
  Code, 2020 — cite that alongside or instead of the 1948 Act if currency matters to your audience.
- **P&ID tags/connectivity were hand-authored**, not extracted by a live computer-vision pass —
  see `scripts/generate_drawings.py` for the reasoning.
- **Graph engine is NetworkX, not Neo4j** — no Docker was available in the build environment. Every
  trigger rule in `graph/rules.py` is written as a small named function shaped like the equivalent
  Cypher query specifically so porting to real Neo4j is mechanical, not a rewrite.
- **No git history before this commit** — the whole build happened in one session; this is the
  first commit, not a curated history.

## The three planted storylines (for anyone reading the corpus cold)

1. **The unclosed loop** — `P-101A`, a 2019 incident recommendation never folded into `SOP-33`,
   echoed by two near-misses, reopened by `WO-2026-4471`.
2. **The cross-equipment pattern** — `E-204`, `E-206`, `E-211`, five incidents over four years,
   same fouling mechanism, never cross-referenced by the engineers who filed them.
3. **The knowledge cliff** — `V-2301`, 14 work orders, 11 signed by R. Krishnan (retiring
   2026-08-15), zero governing procedures.

All three resolve by graph traversal alone — see `tests/acceptance_tests.py` test 3.
